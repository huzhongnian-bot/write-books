# 二创写作平台 — 产品设计（v0.2）

> 基于已有小说进行二次创作的 AI 辅助写作平台
>
> v0.2 变更：新增产品分析、MVP 分期、核心数据模型、AI 管线架构、质量评估与版权合规机制；对 v0.1 待讨论议题给出结论建议。工程侧配套见 [harness-engineering.md](./harness-engineering.md)。

---

## 一、产品定位

面向同人/二创写作者的 AI 协作写作平台。用户上传已有小说作品，平台通过 AI 深度分析原作，提供结构化的二创方向建议，并支持多线叙事设计与对话式内容生成，降低二创写作门槛，提升创作效率。

**目标用户：** 同人写手、网文爱好者、二次创作社区创作者

**核心价值：** 从"读完想写"到"落笔成章"的全流程 AI 辅助

**核心资产洞察（v0.2 新增）：** 这个产品真正的壁垒不是"能生成小说"，而是**结构化的原作知识库（原作百科）+ 用户设定层 + 脚本层构成的"创作上下文引擎"**。生成质量取决于每次生成时喂给模型的上下文质量——所有模块的设计都应围绕"如何把正确的上下文在正确的时机组装出来"展开。

## 二、产品分析（v0.2 新增）

### 优势

- 四大二创方向（续写/支线/平行/融合）是很好的心智框架，把开放式创作收敛成可引导的流程
- "原作百科 → 设定定制 → 脚本 → 生成"的漏斗结构清晰，每一层都为下一层提供结构化上下文

### v0.1 的主要缺口（本版补齐）

| 缺口 | 风险 | 本版对策 |
|------|------|----------|
| 没有 MVP 界定 | 五个模块全量开发，18 个月做不完 | §六 分期路线图 |
| 没有数据模型 | "原作百科/设定/脚本"停留在概念，无法开工 | §四 核心数据模型 |
| 没有 AI 工程设计 | 长篇小说（常见 100–500 万字）远超模型单次上下文，直接灌不进去 | §五 AI 管线架构 |
| 没有质量度量 | "生成得好不好"无法回归验证，prompt 一改全靠肉眼 | §七 质量评估 + harness 文档 |
| 版权只是议题 | 上线即合规风险 | §八 合规机制 |
| 没有成本模型 | 全书分析一次的 token 成本可能高于订阅收入 | §五.4 成本控制 |

### 关键产品判断

1. **原作百科必须可编辑**。全自动抽取一定有错漏，用户修正的过程本身就是创作投入，也让百科成为用户资产（提升留存）。
2. **模块三（可视化脚本编排）是最大的工程黑洞**。拖拽画布、多线交叉可视化的开发成本极高，MVP 阶段用"结构化大纲编辑器"（树形/列表 + 节点属性面板）承载同样的数据模型，画布留到后期。
3. **生成不追求"一键成书"**，而是"每次生成一个场景/段落，用户始终在环内"。这决定了交互形态是对话式工作台，不是批处理。

## 三、核心功能模块

### 模块一：原作导入与智能解析

**目标：** 将小说文件转化为结构化的"创作素材库"

| 能力 | 说明 |
|------|------|
| 文件导入 | 支持 TXT / EPUB 上传（MVP 先做 TXT，编码探测 + 章节切分；PDF 排版噪音大，延后） |
| 基础信息提取 | 自动识别书名、作者、章节结构、字数统计 |
| 背景设定分析 | 提取时代背景、地理环境、社会制度、核心规则（如修炼体系、魔法体系） |
| 情节脉络梳理 | 按章节/卷生成情节摘要，识别主线与支线，标注关键转折点 |
| 人物关系图谱 | 提取主要角色，分析性格特征、角色关系（敌友/师徒/情侣等）、成长弧线 |
| 世界观架构 | 归纳力量体系、势力分布、历史年表、核心设定规则 |

**产出物：** 一份可交互浏览的「原作百科」（Work Bible），每个条目带**出处锚点**（指向原文章节段落），用户可在此基础上标注、修改、补充。

**v0.2 明确：** 解析是"全自动初稿 + 用户校订"模式（回答议题 1）。抽取结果标注置信度，低置信条目引导用户确认。

### 模块二：二创方向建议与自定义设定

四大基础方向不变（续写延展 / 支线展开 / 平行重构 / 跨界融合），v0.2 补充两点：

1. **设定实现为"覆盖层"（overlay），不是复制修改。** 用户的角色改造、设定覆写、时间线调整都存为对原作百科的 diff。生成时"原作百科 + 覆盖层"合并出有效设定。好处：原作百科修正后二创自动受益；一部原作可开多个二创项目；差异一目了然（也服务未来的"原作对比"功能）。
2. **支持自定义方向**（回答议题 2）：四大方向本质是覆盖层的预设模板（续写 = 空覆盖 + 时间线延长；平行重构 = 单点事件改写……）。方向系统实现为模板机制，用户可从空白模板自建。

### 模块三：多线多角色脚本设计

数据模型见 §四。**MVP 形态为结构化大纲编辑器**：剧情线列表 → 每条线是场景节点序列 → 节点面板编辑 POV、参与角色、时间、地点、情节要点、伏笔关联。可视化画布（拖拽、曲线、情感曲线图）为 P2 增强。

**脚本粒度（回答议题 3）：场景级。** 场景是生成的自然单位（一次生成 800–3000 字），段落级过碎（编排成本超过收益），章节级过粗（上下文不够精确）。

### 模块四：对话式小说内容生成

工作模式、五种交互（指令生成/续写/对话引导/局部重写/风格迁移）与生成控制（长度/节奏/详略/角色一致性/伏笔呼应）沿用 v0.1。v0.2 明确工程含义：

- 每次生成是**流式**输出（用户看着文字长出来，可随时打断）
- "角色一致性/伏笔呼应"不是 prompt 里的一句话，而是上下文组装器的职责：自动检索该场景涉及角色的档案、说话风格样例、相关伏笔条目注入上下文（见 §五.2）
- 每个场景保留生成历史（版本链），支持回退与分叉

### 模块五：待定功能

优先级结论（回答议题 6）：

| 功能 | 优先级 | 理由 |
|------|--------|------|
| 版本管理（场景级历史/回滚） | **P0，并入模块四** | 生成-重写工作流的基础能力，不是增值功能 |
| 导出（TXT/EPUB） | **P1** | 闭环最后一步，成本低 |
| AI 角色扮演 | P2 | 复用角色档案即可实现，差异化卖点 |
| 原作对比 | P2 | 覆盖层模型天然支持 |
| 社区协作 / 读者反馈 | P3 | 引入多人权限/内容审核，另一条产品线的复杂度 |

## 四、核心数据模型（v0.2 新增）

```
Project（二创项目）
├── SourceWork（原作）
│   ├── Chapter[]（章节：原文、字数、token 数、序号）
│   └── WorkBible（原作百科）
│       ├── Setting[]      世界观条目（类型/内容/出处锚点/置信度）
│       ├── Character[]    角色（性格、能力、口吻样例、成长弧线、出处）
│       ├── Relationship[] 角色关系（类型、演变、出处）
│       ├── PlotArc[]      情节线（主线/支线、关键转折点→章节锚点）
│       └── TimelineEvent[] 时间线事件
├── Overlay（设定覆盖层）
│   └── Patch[]（针对 Bible 条目的 add / modify / remove + 理由）
├── Script（脚本）
│   └── Storyline[]（剧情线）
│       └── SceneNode[]（场景：POV、参与角色、时地、情节要点、
│                        伏笔引用[]、与其他线的交叉引用[]）
└── Manuscript（成稿）
    └── SceneDraft[]（场景 ↔ SceneNode，含版本链 + 生成会话记录）
```

要点：

- **一切 AI 抽取结果都是带出处、可编辑的结构化数据**，不是一次性的分析报告
- Bible 条目的 JSON Schema 同时是：数据库 schema、抽取时的 structured output schema、评测 golden 数据的 schema——一处定义三处使用（见 harness 文档）
- 存储选型：MVP 用 SQLite（本地单文件，方便 fixture 与测试），Drizzle ORM，schema 用 zod 定义并推导；上线换 Postgres 只换驱动

## 五、AI 管线架构（v0.2 新增）

### 5.1 摄取管线（原作 → 百科）

长篇网文常见 100–500 万字。中文经验值约 1 字 ≈ 1–1.6 token（需用 `count_tokens` 实测），即 150–800 万 token，**远超单次 1M 上下文**，必须分层处理：

```
TXT 上传 → 章节切分 → 逐章 token 计数
   ↓
第一层（并行，Batch API，半价）：逐章结构化抽取
   每章一个请求：情节摘要 + 出场角色/事件/设定线索（structured outputs 强约束 JSON Schema）
   ↓
第二层（归并）：实体归一与合并
   同名角色合并、别名归一、关系去重 —— 代码 + 小模型辅助判定
   ↓
第三层（汇总）：卷级/全书级
   主线支线识别、角色弧线、世界观体系归纳 —— 输入为第一二层的结构化产物（已压缩数百倍）
   ↓
写入 WorkBible，低置信条目标记待用户确认
```

- 逐章抽取用 **Message Batches API**（异步、50% 折扣、24h 内完成），摄取本来就是后台任务，用户上传后看进度条
- 全流程记录为可恢复的 job（队列表 + 状态机），单章失败可重试而不重跑全书

### 5.2 生成管线（脚本节点 → 正文）

每次生成一个场景，核心是**上下文组装器（context assembler）**：

```
[system prompt：写作规范 + 平台指令（冻结，缓存）]
[原作百科核心 + 有效设定（Bible⊕Overlay 合并，项目内稳定，缓存断点）]
[本场景相关上下文（动态检索）：
   参与角色档案 + 口吻样例 / 相关伏笔条目 / 关联场景脚本 / 原文风格样例片段]
[前文正文：最近 1–2 个场景全文 + 更早场景的摘要链]
[用户本轮指令]
```

- **Prompt caching 是成本命门**：按稳定性排序内容，稳定前缀（system + 合并设定）打 `cache_control` 断点，会话内命中率应 >90%（缓存读取 ≈ 0.1× 输入价）。注意静默失效点：设定一旦被用户编辑，缓存重建是预期行为；但绝不能在前缀里插时间戳/随机 ID
- **流式输出**（SSE 到前端），长生成不超时；开 adaptive thinking 提升情节连贯性
- 生成后异步跑**一致性检查器**（轻量请求）：角色行为是否违背档案、是否触碰设定禁区，结果以批注形式提示用户

### 5.3 模型选型

| 环节 | 默认模型 | 说明 |
|------|----------|------|
| 正文生成、对话引导、全书汇总 | `claude-opus-4-8`（$5/$25 每百万 token） | 长文写作与文学性最优的默认选择；追求极致可评估 `claude-fable-5`（$10/$50，注意其 refusal 处理与 30 天数据保留要求） |
| 逐章批量抽取 | `claude-opus-4-8` 起步 | 量大后可用评测集对比 `claude-haiku-4-5`（$1/$5）的抽取质量，达标再降——**用数据决策，不拍脑袋**（评测机制见 harness 文档） |
| 一致性检查、实体归一辅助 | `claude-haiku-4-5` | 判定类轻任务 |

所有调用统一走 `src/lib/ai/` 封装层（TypeScript SDK `@anthropic-ai/sdk`），模型名、prompt、schema 都不散落在业务代码里——这是 harness 的挂载点。

### 5.4 成本控制

以 200 万字小说（约实测 300 万 token）为例的量级估算（需实测校准）：

- 全书摄取：输入 ~300 万 token + 输出 ~30 万 token，Batch 半价后约 **$11–12/本**，一次性成本
- 单场景生成：缓存命中下每次约 3–5 万 token 输入（其中 90% 走缓存读取）+ 2–4k 输出，约 **$0.1–0.2/次**

含义：摄取按"本"收费或限额；生成按次/订阅额度计量。所有请求记录 usage 落库，做成本看板。

## 六、MVP 分期路线（v0.2 新增）

| 阶段 | 范围 | 验证的假设 |
|------|------|------------|
| **P0 · 走通管线**（内部可用） | TXT 导入 → 摄取管线 → 原作百科浏览/编辑；单线脚本（大纲编辑器）；场景级对话式生成（指令生成 + 续写 + 重写）；场景版本链 | 结构化百科 + 上下文组装真的能显著提升生成质量吗？ |
| **P1 · 产品闭环** | 四大方向模板 + 覆盖层设定；多线脚本与交叉标注；风格迁移/对话引导；导出 TXT/EPUB；账号与项目管理 | 目标用户愿意为这个工作流付费吗？ |
| **P2 · 增强** | 可视化脚本画布；AI 角色扮演；原作对比；EPUB 导入 | 留存与差异化 |
| **P3 · 社区** | 协作、读者反馈、发布渠道 | 平台化 |

P0 不做：登录（本地单用户）、支付、EPUB/PDF 解析、可视化画布。

## 七、生成质量把控（回答议题 4）

质量不能靠感觉，建立三层机制（实现细节见 harness 文档 §4）：

1. **抽取质量**：golden 评测集——人工标注若干公版章节的期望抽取结果，每次改 prompt/换模型跑评测，出准确率/召回率
2. **生成质量**：LLM-as-judge 评分卡（角色一致性、设定符合度、文风贴合度、情节推进有效性，1–5 分）对固定场景集打分，作为回归基线
3. **线上兜底**：一致性检查器批注 + 用户重写率/采纳率埋点作为真实质量信号

## 八、版权合规（回答议题 5）

- **定位为私人创作工具**：原作文本仅用于该用户的分析与创作上下文，不公开展示原文、不提供原文下载、不跨用户共享分析结果
- 上传时用户确认"拥有合法阅读来源、二创内容自负其责"；导出文件附带二创声明模板
- 敏感能力隔离：任何"发布/分享"功能（P3）必须走独立合规评审，MVP 不碰分发
- 数据侧：原文加密存储，删除项目即物理删除原文

## 九、用户分层（回答议题 7）

结论：**分层做在引导密度上，不做在功能上**。新手走"向导模式"（方向模板 → 逐步表单 → 生成），熟手直接暴露百科/脚本/生成三个工作台。同一套数据模型，两层壳。

## 十、技术架构映射（当前仓库）

现状：Next.js 16.2 + React 19 + Tailwind 4 + shadcn/ui（base-nova）+ TS strict，纯前端脚手架。

| 层 | 选型 | 位置 |
|----|------|------|
| UI | 现有 shadcn 组件 + `/design` 工作台（见 harness 文档） | `src/components/`, `src/app/(workspace)/` |
| API | Route Handlers（`app/api/*/route.ts`），生成接口返回 SSE 流 | `src/app/api/` |
| 变更操作 | Server Actions（表单/编辑类） | 就近 co-locate |
| AI 层 | `@anthropic-ai/sdk` 封装：client、prompt 注册表、上下文组装器、mock 开关 | `src/lib/ai/` |
| 数据 | SQLite + Drizzle（zod schema 单一来源），P1 迁 Postgres | `src/lib/db/` |
| 后台任务 | 摄取 job：队列表 + 状态机，开发期用 Node 常驻 worker 轮询 Batch API | `src/lib/jobs/` |
| 密钥 | `ANTHROPIC_API_KEY` 仅存在于服务端环境变量 | `.env.local` |
